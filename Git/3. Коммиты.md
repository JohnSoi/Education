Контроль версий автоматически не сохраняет все изменения в файле - это бы занимало невероятные объемы данных. Поэтому для сохранения состояний используется коммит - своего рода скриншот текущего состояния в [репозитории](1.%20Репозиторий.md). Так же в промер можно привести контрольные точки в восстановлении системы Windows. 
Между этими сохраненными состояниями можно переключаться и видеть то состояние репозитория, в котором оно находилось.

> **Как хранятся изменения в системе контроля версий?**
> 
> Хранить файл целиком и вычислять изменения на лету очень накладно. Поэтому в системе контроля версий хранится только разница (все изменения в файле). Это называется дельтой. Например, если мы создаем файл, то дельтой будет появление файла (при удалении - его отсутствие), а если же мы добавляем или изменяем файл - то дельтой будут измененные строки.

Для того, чтобы сделать коммит, достаточно добавить файл в [отслеживаемые](2.%20Состояния%20файлов%20в%20системе%20контроля%20версий.md) и ввести команду:
```bash
git commit -m "Init project"
```

```"Init project"``` - это текст комментария и пишется в свободной форме. Он предназначен для описания произведенных действий для последующей удобной навигации и отслеживания.
В результате вы увидите примерно следующее:
![Сообщение о коммите](./Материалы/Сообщение%20коммита.png)
Мы видим следующее:
* ```master``` - название ветки
* ```root-commit``` - признак корневого коммита. Самый первый коммит в репозитории. Он не ссылается ни на кого, но последующий коммит будет ссылаться на этот.
* ```f4b1376``` - уникальный номер коммита (часть хеша (уникального идентификатора коммита)). По нему можно переходить к этому коммиту. **У вас он будет другой, так как генерируется рандомно.**
